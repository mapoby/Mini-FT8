# Testing

Mini-FT8 is embedded firmware for the M5Stack Cardputer ADV — there is no `npm test`, no CI-run
unit test suite in the JavaScript sense, and no test framework like Jest or pytest. Testing here
covers two distinct layers: a small host-native C/C++ test suite (`tests/tx_e2e/`) that validates
the FT8/FT4 protocol encode/decode path and TX timing logic on your development machine, and
manual hardware verification (flash to a real Cardputer ADV + radio, observe behavior) for
everything that depends on actual USB audio, CAT control, or RF.

## Test Framework and Setup

There is no JavaScript/npm test runner. The host-native suite in `tests/tx_e2e/` is a standalone
CMake project (`tests/tx_e2e/CMakeLists.txt`) built independently of the ESP-IDF firmware build.
It compiles `ft8_lib` (the FT8/FT4 protocol library) as a native static library, links each test
against it, and runs the resulting binaries via CTest.

**Requirements:**
- CMake >= 3.16
- A native C/C++ toolchain — **MinGW/GCC or Clang only**. MSVC is explicitly unsupported:
  `tests/tx_e2e/CMakeLists.txt` fails the configure step with `MSVC is unsupported due to its
  lack of C99 VLA support. Please use MinGW/GCC or Clang.`
- On 32-bit x86 GCC (MinGW), the build forces `-mfpmath=sse -msse2` to avoid x87 80-bit extended
  precision breaking `kiss_fftr`'s even-FFT-size requirement (see the comment at the top of
  `tests/tx_e2e/CMakeLists.txt`).

**Setup:**

```bash
cd tests/tx_e2e
cmake -S . -B build -G "MinGW Makefiles"   # or "Unix Makefiles" on Linux/macOS
cmake --build build
```

## Running Tests

Run the full suite via CTest from the build directory:

```bash
cd tests/tx_e2e/build
ctest --output-on-failure
```

The suite registers seven tests (`tests/tx_e2e/CMakeLists.txt`):

| Test | Source | Validates |
|---|---|---|
| `l1_encoder` | `test_l1_encoder.cpp` | FT8/FT4 message encode → synth → decode round trip (Layer 1, no state machine) |
| `l2_state_machine` | `test_l2_state_machine.cpp` | TX state machine (`tx_state_machine.cpp/h`) against decoded audio output |
| `l3_poll_timing` | `test_l3_tx_poll_timing.cpp` | Reproduces the main-loop polling bug where FT4's 48 ms symbol period is not a multiple of a 10 ms poll interval, causing tone jitter and LDPC decode failure; asserts FT8 (160 ms period) is unaffected |
| `l4_timer_isolation` | `test_l4_tx_timer_isolation.cpp` | Confirms that driving `tx_tick()` from an `esp_timer` callback (ISOLATED) rather than the main UI loop (COUPLED) keeps FT4 TX timing accurate even when UI drawing (e.g. waterfall rendering) blocks the main loop |
| `golden_rx` | `test_golden_rx.cpp` | Decodes committed WAV files in `tests/tx_e2e/golden/` and asserts each produces its expected message — validates the decoder independently of the encoder |
| `ta_format` | `test_ta_format.cpp` | QMX `TA%04d.%02d;` audio-tone-frequency command formatting (`qmx_set_tone_hz()` / `tx_send_ta()`), including the floor-vs-round negative-fraction bug fix |
| `kh1_tone_map` | `test_kh1_tone_map.cpp` | KH1 `FOxx` tone-offset mapping from FT8/FT4 tone index to KH1 command values |

Run a single test directly:

```bash
ctest --output-on-failure -R golden_rx
```

or run the compiled binary directly (some binaries, like `test_wav_diag`, are diagnostic tools
rather than CTest-registered assertions and must be run this way):

```bash
./Release/test_golden_rx   # Windows (MinGW/Visual Studio generator output layout may vary)
./test_golden_rx           # Linux/macOS
```

`test_wav_diag` is not registered as a CTest case — it is a standalone diagnostic tool for
analyzing a WAV file that fails to decode:

```bash
test_wav_diag ../golden/ft4_cq_w1xyz_fn42.wav "CQ W1XYZ FN42" ft4
```

## Golden WAV Regression Files

`tests/tx_e2e/golden/` holds committed WAV fixtures used by `golden_rx` (see
`tests/tx_e2e/golden/MANIFEST.txt` for the full list of files, their protocol, expected message
text, and base tone frequency). Files are 16-bit mono PCM at 6000 Hz (the firmware's native
processing rate), generated by the `gen_golden` executable using the same encode → synth path as
`test_l1_encoder`. `MANIFEST.txt` also documents a device-recorded 12 kHz file,
`ft4_cq_w6dvn_dm04.wav` (with a note that the harness auto-detects sample rate and
re-initializes the monitor accordingly), but that file is not currently present in
`tests/tx_e2e/golden/` — only 7 of the 8 manifest-listed WAVs exist on disk as of this writing.

To regenerate golden WAVs after a deliberate encoder, synth, or sample-rate change:

```bash
cd tests/tx_e2e/build
./Release/gen_golden.exe ../golden   # Windows
./gen_golden ../golden               # Linux/macOS
```

Then commit the updated WAV files.

## Writing New Tests

- Test files follow the naming convention `test_<name>.cpp` in `tests/tx_e2e/`.
- Add a new `add_executable(...)` and, if it asserts pass/fail rather than being a diagnostic
  tool, a matching `add_test(NAME ... COMMAND ...)` entry in `tests/tx_e2e/CMakeLists.txt`.
- `decode_helper.cpp/h` provides shared decode assertion helpers used across most tests.
- `synth.c/h` provides the reference FSK synthesizer used to generate test audio from encoded
  symbols.
- `tx_state_machine.cpp/h` and `timing_sim.h` provide reusable TX timing/state-machine simulation
  used by the L2–L4 tests.
- Tests link against `ft8_lib` built from `components/ft8_lib/` sources directly (the non-IDF
  portable subset: `ft8/`, `fft/`, `common/`) — no ESP-IDF or FreeRTOS dependency is available in
  this suite.

## Coverage Requirements

No code coverage tooling (gcov/lcov, `--coverage` flags) is configured for `tests/tx_e2e/`. No
coverage threshold is enforced.

## CI Integration

No `.github/workflows/` directory exists in this repository. The `tests/tx_e2e/` suite is
**not** run automatically in CI — it must be built and run manually on a development machine
before submitting changes that touch the protocol encode/decode path, TX timing, or CAT command
formatting.

## Hardware Verification

Because Mini-FT8 is firmware for a specific ESP32-S3 board (M5Stack Cardputer ADV) driving real
radios (QMX, QDX, KH1, and — per this milestone — the Yaesu FTX-1) over USB CAT and USB/serial
audio, most functional verification does **not** happen through the automated `tests/tx_e2e/`
suite. That suite validates the pure protocol/timing logic in isolation on a host machine; it
cannot exercise USB Host driver behavior, actual radio CAT responses, real audio codec timing, or
RF propagation.

The rest of the system is verified manually against physical hardware:

1. Build and flash firmware to a real M5Stack Cardputer ADV (see `build.ps1`,
   `build_and_flash.bat`, or `build_and_flash.py` in the project root, and
   `MiniFT8_Merged_Auto.bin` as the merged flash artifact).
2. Connect the target radio (QMX, QDX, KH1, or FTX-1) over USB/serial as applicable.
3. Observe runtime behavior over the debug UART / serial console and the device's own LCD UI
   (RX decode list, waterfall, TX queue, STATUS mode) to confirm CAT sync, audio capture/playback,
   and correct FT8/FT4 TX timing on-air.
4. For protocol-level regressions specifically (e.g. a TX audio glitch or decode failure suspected
   on a real recording), capture the WAV from the device and run it through `test_wav_diag` or add
   it to `tests/tx_e2e/golden/` for `golden_rx` coverage — this is the bridge between hardware
   observations and the host-native automated suite.

This means changes to USB Host driver integration (e.g. the FTX-1's `usb_host_cp210x_vcp`
component, or `stream_uac.cpp` audio pipeline changes), CAT protocol handling, or timing-sensitive
code that only manifests on real hardware require a physical flash-and-observe checkpoint in
addition to (not instead of) the `tests/tx_e2e/` suite passing.
