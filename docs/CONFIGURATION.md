# Configuration

Mini-FT8 is embedded firmware for the M5Stack Cardputer ADV — there is no server process, no
`.env` file, and no environment variables. "Configuration" here covers three distinct layers:

1. **Build-time configuration** — CMake options and compiler defines that change what code is
   compiled into the firmware image (set once, at build time).
2. **Flash layout** — `partitions.csv`, which defines how the ESP32-S3's flash is partitioned
   between the app image, NVS, and the on-device filesystem.
3. **Runtime configuration** — `Station.txt`, a plain-text `key=value` file persisted to the
   internal FATFS partition. This is the closest analog to a `.env` file: it holds the operator's
   callsign, grid square, radio selection, band frequencies, and other settings. It is normally
   edited through the on-device menu UI, not hand-edited, but the format is human-readable.

## Build-time configuration

### CMake options

| Option | Default | Effect |
|---|---|---|
| `ENABLE_FT4` | `ON` | When on, FT4 protocol code is compiled in and the user can switch between FT8 and FT4 at runtime via Settings → Mode. When off, the firmware builds FT8-only and the mode-switch menu item is omitted. |

Set at build time via:

```bash
idf.py build -DENABLE_FT4=OFF
```

Source: `CMakeLists.txt` (root), which translates the option into `add_compile_definitions(ENABLE_FT4=...)`.
The compiled-in default is also mirrored in `main/feature_flags.h` (`#define ENABLE_FT4 1` if not
otherwise defined by CMake).

### Compiler flags

`main/CMakeLists.txt` builds the `main` component with `-Werror`, so any compiler warning in
`main/` fails the build.

### ESP-IDF / sdkconfig

The project uses ESP-IDF v5.5.1 with a checked-in `sdkconfig` (auto-generated by `idf.py
menuconfig` / `idf.py build`, not hand-authored). Notable non-default settings:

| Setting | Value | Purpose |
|---|---|---|
| `CONFIG_ESPTOOLPY_FLASHSIZE` | `8MB` | Target board (Cardputer ADV) has 8 MB of flash |
| `CONFIG_PARTITION_TABLE_CUSTOM` | `y` | Uses the custom `partitions.csv` table instead of an ESP-IDF stock table |
| `CONFIG_PARTITION_TABLE_CUSTOM_FILENAME` | `partitions.csv` | Points to the project's partition table |
| `CONFIG_BT_ENABLED` | not set | Bluetooth is disabled |
| `CONFIG_FATFS_VOLUME_COUNT` | `2` | Two FATFS volumes mounted (internal flash + SD card) |

## Flash partition layout

Defined in `partitions.csv`, offset by ESP-IDF's `CONFIG_PARTITION_TABLE_OFFSET=0x8000`:

| Name | Type | SubType | Offset | Size |
|---|---|---|---|---|
| `nvs` | data | nvs | `0x9000` | `0x6000` (24 KB) |
| `phy_init` | data | phy | `0xf000` | `0x1000` (4 KB) |
| `factory` | app | factory | `0x10000` | `0x4F0000` (~5 MB) |
| `fatfs` | data | fat | `0x500000` | `0x300000` (3 MB) |

- `nvs` — non-volatile storage for low-level driver/system state (not `Station.txt`; see below).
- `factory` — the single application image (no OTA slots configured).
- `fatfs` — internal FAT filesystem holding `Station.txt`, ADIF QSO logs (`[YYMMDD].adi`), and
  RX/TX transcript logs (`RT[YYMMDD].txt`).

The build produces a merged flashable image via `esptool --chip esp32s3 merge_bin`, output as
`MiniFT8_Merged_Auto.bin` (see the `POST_BUILD` step in `CMakeLists.txt`). First boot after
flashing a new partition table formats the 3 MB FATFS partition if it isn't already formatted.

## Runtime configuration: Station.txt

`Station.txt` lives on the internal FATFS partition and is read/written by
`load_station_data()` / `save_station_data()` in `main/main.cpp`, via the storage abstraction in
`components/storage_service/`. It is a flat `key=value` text file, one setting per line. Every
settings mutation made through the Cardputer's on-device menu UI calls `save_station_data()`,
which rewrites the whole file atomically (`storage_file_write_atomic`).

### Settings written by `save_station_data()`

| Key | Format | Description |
|---|---|---|
| `band0`…`bandN` (or `ft4_band0`…`ft4_bandN`) | `band<idx>=<freq_kHz>` | Per-band dial frequency in kHz. FT8 and FT4 use separate key prefixes (`band` vs `ft4_band`) so switching protocol doesn't overwrite the other mode's frequencies. |
| `offset` | int | TX audio offset in Hz |
| `band_sel` | int | Index of the currently selected band |
| `date` | string | Saved UTC date |
| `time` | `HH:MM:SS` | Saved UTC time |
| `cq_type` | int (`CqType` enum) | CQ prefix variant: `CQ`, `CQSOTA`, `CQPOTA`, `CQQRP`, `CQFD`, `CQFREETEXT` |
| `cq_ft` | string | Free-text CQ message |
| `skiptx1` | 0/1 | Skip TX1 (first transmission) toggle |
| `free_text` | string | Manual free-text TX message |
| `call` | string | Operator callsign |
| `grid` | string | Manually-entered grid square (Maidenhead) |
| `offset_src` | int (`OffsetSrc` enum) | How the TX audio offset is chosen: `RANDOM`, `CURSOR`, `RX` |
| `radio` | int (`RadioType` enum) | Selected radio backend — see table below |
| `gps_baud` | int | GPS UART baud rate (9600 or 115200) |
| `gnss_lora` | 0/1 | GNSS-over-LoRa GPS source enabled |
| `comment1` | string | Free-form comment field |
| `ignore_prefixes` | string | Callsign prefixes to ignore in decode list |
| `rxtx_log` | 0/1 | Enable RX/TX transcript logging to `RT[YYMMDD].txt` |
| `active_bands` | string | Active band list used for multi-band scanning |
| `rtc_sleep_epoch` | int64 | Epoch time saved before deep sleep, for RTC drift compensation |
| `rtc_comp` | int | RTC drift compensation value |
| `autoseq_max_retry` | int | Max retry count for the autoseq QSO state machine |
| `protocol_mode` | `FT4` (only written when pending) | Requested protocol mode; takes effect on next reboot (`ENABLE_FT4` builds only) |

Note: `beacon` (CQ beacon mode) is intentionally **not** persisted — it always resets to `OFF` on
load, regardless of what was active when the device was last powered off.

### `radio=` values (`RadioType` enum, `main/station_types.h`)

| Value | Radio |
|---|---|
| `0` | NONE |
| `1` | TRUSDX |
| `2` | QMX (default / fallback for unrecognized values) |
| `3` | KH1_USBC (also accepted as legacy alias `KH1`) |
| `4` | KH1_MIC |
| `5` | QDX |
| `6` | FTX1 |

`parse_radio_config_value()` also accepts the string tokens `KH1`, `KH1-USBC`, `KH1_USBC`,
`KH1USB`, `KH1-MIC`, `KH1_MIC`, `KH1MIC`, `QDX`, `FTX1`, `FTX-1` (case-insensitive) for the
`radio=` line, in addition to the plain integer form. Any unrecognized value falls back to QMX.

### Required vs. optional settings

There are no settings that block firmware startup if `Station.txt` is missing or unreadable —
`load_station_data()` returns early with hardcoded defaults (see next section) if the file can't
be opened, and the firmware proceeds in that default state. All settings are therefore optional
in the sense that the device boots either way; callsign and grid should be set by the operator
before making QSOs, but nothing enforces this at boot.

### Defaults when `Station.txt` is absent or a key is missing

Set at the top of `load_station_data()` in `main/main.cpp` before the file is parsed:

| Setting | Default |
|---|---|
| `rtc_comp` | `kRtcCompFixed` (compiled-in constant) |
| `autoseq_max_retry` | `AUTOSEQ_MAX_RETRY` (compiled-in constant) |
| `gps_baud` | `115200` |
| `gnss_lora` | `false` |
| `beacon` | always forced `OFF` regardless of file contents |
| `radio` (in-memory global `g_radio`) | `RadioType::QMX` |

Band frequency defaults for FT4 mode (used when `protocol_mode=FT4` is detected and no per-band
`ft4_band<N>=` override exists) are hardcoded in `load_station_data()`:

### Per-environment overrides

Not applicable — Mini-FT8 has no concept of separate development/staging/production
environments. The only "environment" distinction that exists is:

- **Build-time**: `ENABLE_FT4` (FT8-only vs. FT8+FT4 firmware build).
- **Runtime**: `protocol_mode=FT4` in `Station.txt`, which takes effect on next reboot and swaps
  the active band-frequency defaults and per-band storage key prefix (`band` vs `ft4_band`).
