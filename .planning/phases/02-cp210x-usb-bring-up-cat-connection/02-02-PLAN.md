<objective>
Open the FTX-1's CP210x CAT-1 virtual COM port over USB, deassert RTS/DTR immediately after
open as defense-in-depth against double-PTT, and hard-gate the existing QMX CDC-ACM scan so it
can never misclaim the FTX-1's vendor-class interface. Wire `radio_control_ftx1.cpp`'s `ready()`
to report the real connection state.

Purpose: This is the phase's core transport work — CAT-01 (open), CAT-02 (never-misclaim),
CAT-03 (RTS/DTR deassert). It builds directly on the profile/backend enums from Plan 01.
Output: A new `cp210x_try_open()` function mirroring `cdc_try_open()`'s shape (minus the
hint-scan/blind loop), profile-gated dispatch at both call sites, the CDC-ACM install gate
broadened to include the FTX-1 profile, and `ftx1_ready()` delegating to `cat_cdc_ready()`.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| USB device -> host CDC-ACM/CP210x scan | The FTX-1 is a composite device exposing vendor-class (0xFF) CP210x interfaces plus UAC; the QMX blind interface scan must never claim one of them |
| CP210x port open -> RTS/DTR line state | On open, an un-deasserted control line can key the radio (double-PTT); this boundary is the CAT-03 control point |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-03 | Elevation | `cdc_try_open()` blind interface scan | mitigate | Add `if (s_profile != UAC_PROFILE_QMX) return;` guard (Task 2) — the profile-gating logic is the key control preventing a misclaimed/undefined interface open when the FTX-1 profile also has the CDC-ACM base driver installed |
| T-02-04 | Denial of Service | CP210x port open (RTS/DTR) | mitigate | Call `cdc_acm_host_set_control_line_state(handle, false, false)` immediately after open (Task 1, CAT-03) so the radio cannot be spuriously keyed on connect; hardware-verified in Plan 03 |
| T-02-05 | Tampering | `cp210x_vcp_open()` descriptor walk | accept | Descriptor/vendor-request handling is inside the vendored, bounds-checked `usb_host_cp210x_vcp`/`usb_host_cdc_acm` components; no custom parsing added here |
| T-02-06 | Denial of Service | future CAT read path (`data_cb`) | transfer | `in_buffer_size`/`data_cb` are placeholder-only in Phase 2; input-length/range validation of CAT replies is deferred to Phase 3 (flagged, not ignored) |
</threat_model>
