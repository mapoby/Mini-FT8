<objective>
Add the `espressif/usb_host_cp210x_vcp` build dependency and introduce the FTX-1's own
profile/backend enum values so FTX-1 stops silently aliasing the QMX audio profile. Fix the
Phase-1 placeholder in `get_radio_profile_binding()` that routes FTX-1 through
`AUDIO_SOURCE_QMX_UAC` — the reachable latent bug that today drives FTX-1 into QMX's UAC
profile and exposes the exact CDC-ACM misclaim risk CAT-02 exists to prevent.

Purpose: This plan lays the enum/routing foundation. It does NOT open the port yet (Plan 02).
Its job is to make FTX-1 resolve to a distinct profile so the port-open dispatch in Plan 02 has
a profile to branch on, and to close the routing half of the CAT-02 misclaim exposure.
Output: Component dependency wired into the build; `UAC_PROFILE_FTX1` and
`AUDIO_SOURCE_FTX1_CP210X` enum values; FTX-1 routed to its own backend/profile end to end.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Radio-type selection -> profile/backend resolution | A wrong mapping here is the root cause of the CAT-02 misclaim: it decides which USB open path runs for the FTX-1 |
| USB device -> host | A plugged-in FTX-1 is physically trusted hardware, but its vendor-class (0xFF) CP210x interfaces must never be routed into the QMX CDC-ACM scan |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-01 | Elevation/Tampering | `get_radio_profile_binding()` FTX1 case | mitigate | Route FTX-1 to `AUDIO_SOURCE_FTX1_CP210X`/`UAC_PROFILE_FTX1` (Task 2) so it never enters the QMX profile path that would run the blind CDC-ACM interface scan against the FTX-1's vendor-class interface |
| T-02-02 | Tampering | USB descriptor parsing | accept | Descriptor parsing is performed by the vendored, bounds-checked `usb_host_cdc_acm`/`usb_host_cp210x_vcp` components; no custom descriptor-walking code is added in this plan |
</threat_model>
