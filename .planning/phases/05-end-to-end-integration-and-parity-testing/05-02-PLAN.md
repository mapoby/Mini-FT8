<objective>
Prove that a complete FT4 QSO works end-to-end on the physical FTX-1, using the same Phase 1-4 stack already proven for FT8 in plan 05-01, but under FT4's tighter 7.5-second slot cadence (half of FT8's 15 seconds). This is the second and final hardware checkpoint of Phase 5, and closes out the milestone's E2E validation: after this plan, both E2E-01 and E2E-02 are confirmed and ROADMAP Success Criterion 3 (sustained multi-cycle operation, no regressions) is verified across both protocols.

No new code is expected. FT4 uses the same `ProtocolConfig` abstraction, autoseq engine, CAT backend, and audio pipeline as FT8 -- only the slot period constant differs (`g_protocol` selects FT4 timing at boot from Station.txt, per STATE.md/ARCHITECTURE.md's protocol-selection pattern). The specific risk this plan checks for for is timing headroom: whether the CAT drain-before-unkey (PTT-02) and audio drain (`uac_tx_end()`) sequence -- both already proven correct for FT8's 15s slots in 03-02/04-02 -- still complete comfortably inside FT4's 7.5s slot boundary without clipping the next slot's RX window.

Claude's Discretion (explicitly flagged, not silently assumed): as in 05-01, either a real on-air FT4 contact or a loopback/two-radio setup satisfies E2E-02. Record the method and rationale used, independently of whichever method 05-01 used for FT8 -- they do not have to match.

If this checkpoint surfaces a genuine code defect (not a methodology/environment issue), it must be handled as an explicit gap-closure fix task in a follow-up plan, not worked around silently.

Purpose: Validate E2E-02 and complete ROADMAP Success Criterion 3 (sustained multi-cycle, no regressions across both FT8 and FT4) against real hardware, closing out the v1 milestone.
Output: 05-02-SUMMARY.md recording FT4 QSO evidence, FT4 timing-headroom findings, and the final consolidated Phase 5/milestone regression verdict.
</objective>
