<objective>
Add the Yaesu FTX-1 as a fourth radio backend, selectable in station configuration, with a complete but stubbed `radio_control_ops_t` vtable that mirrors the QMX/QDX/KH1 pattern exactly. No hardware is required — `ready()` returns `false` and every other hook returns `ESP_ERR_INVALID_STATE`.

Purpose: Establishes the code-structure foundation (enums, dispatch, menu, persistence, build registration) that Phases 2-4 build real CP210x CAT and USB audio on top of. This is a pure mechanical pattern extension with zero new dependencies.

Output: A new `main/radio_control_ftx1.cpp` stub backend plus additive edits to 6 existing files, all compiling under `-Werror`, so a user can cycle the config menu to "FTX-1" without affecting existing radios.
</objective>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|

This phase extends that allow-list with one value (6); it does not add a new parsing path. |

No network boundary, no external device I/O, and no user-supplied free-form input are introduced. `ready()` returns `false`, so the FTX-1 backend performs no CAT or USB activity in this phase.

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-01 | Tampering | Station.txt `radio=` value | accept | A corrupted/out-of-range int already falls through the existing `default: return RadioType::QMX;` guard — safe default, unchanged by this phase. Adding value 6 to the allow-list does not weaken the guard. |
| T-01-02 | Denial of Service | FTX-1 stub ops (`ready()`=false) | accept | Stubs return `ESP_ERR_INVALID_STATE` with no allocation, blocking, or recursion; no reachable path can hang or crash. Verified by build + human boot check (Task 3). |
| T-01-03 | Elevation of Privilege | new backend dispatch case | accept | Adds one `switch` case returning a static const struct pointer; no privilege, no memory write, no dynamic dispatch beyond existing pattern. No attack surface. |

Rationale: This is a local embedded firmware code-structure change with no network exposure, no cryptography/auth/session surface, and no new untrusted input path. ASVS L1 V5 (Input Validation) is the only applicable category and is already satisfied by the pre-existing allow-list parser, which this phase extends rather than modifies. Overall: low/no attack surface.
</threat_model>
