## Symptoms

- **Expected behavior:** TX (FSK tone) audio generated by the DDS renderer should reach the FTX-1's
  USB speaker/audio-input interface and be transmitted as a clean FT8/FT4 signal (AUDIO-02).
- **Actual behavior:** The speaker interface fails to open on every single boot this session
  (confirmed across at least 6 separate boot cycles in the captured debug UART log). PTT keys/unkeys
  correctly via CAT and the DDS renderer completes its tone sequence in software, but the audio
  samples have nowhere to go — no actual signal reaches the FTX-1.
- **Error messages:**
  ```
  E (7523) uac-host: uac_host_device_start(2357): No suitable alt setting found
  E (7523) UAC_STREAM: spk pre-start failed at enum: ESP_ERR_NOT_FOUND
  ```
  Recurs identically on every boot (timestamps 7523/11678/21925/38269/42948/etc. across different
  sessions — always within the first ~10-40s after enumeration).
- **Timeline:** This exact failure was already observed (but not root-caused) during Phase 4's
  04-02 hardware checkpoint, where it was tentatively attributed to "D-05: if the FTX-1's speaker
  doesn't accept 24-bit/stereo/48kHz, that is a hardware-validation finding... not something to
  pre-build speculatively now" (04-CONTEXT.md). It recurred identically during this Phase 5 session,
  now with a working debug-UART capture that lets us see the actual USB Audio descriptor dump for
  the first time.
- **Reproduction:** Boot the Cardputer with the FTX-1 selected as active radio and connected. The
  speaker (UAC-OUT, TX_CONNECTED event) pre-open always fails at enumeration, on every boot.

## Evidence

- timestamp: 2026-07-07
  checked: Debug UART log captured live during Phase 5 hardware checkpoint (this session).
  found: USB device enumeration for the FTX-1's internal audio codec (addr:3, VID:0x0d8c PID:0x0016
    — a common C-Media-style USB Audio chip) shows exactly two audio streaming interfaces:
    IF1 (alt1, 1 endpoint, OUT direction — classified "Speaker captured: addr=3 iface=1" by the
    driver) and IF2 (alt1, 1 endpoint, IN direction — "Microphone connected - addr:3, iface:2").
    The mic interface's descriptor dump (only descriptor dump the driver ever prints, since the
    mic candidate-scan loop triggers it on its own failed/retried opens) shows:
    `channels = 2`, `bit_resolution = 16`, `sample_freq: 48000, 44100`. No equivalent dump is ever
    printed for the speaker interface (IF1) because the speaker code path only ever tries ONE fixed
    candidate (2ch/24-bit/48000Hz, matching dds_render_24bit_stereo()'s fixed output format,
    per Phase 4's D-04 decision) and fails immediately with "No suitable alt setting found" before
    any candidate-scan/retry logic would print descriptor details.
  implication: The FTX-1's actual USB audio codec (the same physical chip serving both mic and
    speaker directions, per typical C-Media-style codec design) almost certainly only supports
    16-bit resolution, not 24-bit, symmetric across both directions — consistent with the mic's
    confirmed 16-bit-only capability (candidate 1/4 at 24-bit also fails on the mic path; only
    candidate 2/4 at 16-bit succeeds, per the mic's own candidate-scan log). The speaker's single,
    non-negotiable 24-bit candidate is being rejected for the same underlying hardware reason.
  verified_by: main/stream_uac.cpp TX_CONNECTED handler (search for `scfg.bit_resolution` /
    `FLAG_STREAM_SUSPEND_AFTER_START`, currently ~line 748) constructs exactly one
    `uac_host_stream_config_t` with channels=2, bit_resolution=24, sample_freq=48000 (matching
    `dds_render_24bit_stereo()` in main/dds_q15.cpp, which unconditionally emits 3-byte-per-channel
    24-bit-in-32-bit-container samples). There is no fallback candidate and no alternate renderer.

## Resolution

root_cause: |
  The FTX-1's USB audio codec (C-Media-style chip, VID:0x0d8c PID:0x0016) only supports 16-bit
  sample resolution on its audio streaming interfaces (confirmed via the mic interface's USB Audio
  descriptor dump: channels=2, bit_resolution=16, sample_freq={48000,44100} — no 24-bit option
  exists at all). Mini-FT8's TX path (main/stream_uac.cpp's TX_CONNECTED handler +
  main/dds_q15.cpp's dds_render_24bit_stereo()) only ever constructs and requests a single, fixed
  2ch/24-bit/48000Hz uac_host_stream_config_t with no fallback, so uac_host_device_start() always
  fails with "No suitable alt setting found" / ESP_ERR_NOT_FOUND, and no TX audio is ever rendered
  to real USB frames. This was anticipated and explicitly deferred in Phase 4 (CONTEXT.md D-04/D-05)
  pending hardware validation, which has now confirmed the gap.
fix: |
  1. main/dds_q15.h/.cpp: added dds_render_16bit_stereo() sibling to dds_render_24bit_stereo(),
     sharing the same CPFSK symbol-timing state machine; only per-sample packing differs (native
     int16_t, 2 bytes/channel little-endian, no upshift, via new emit_stereo_frame_16() /
     emit_silence_16() helpers).
  2. main/stream_uac.cpp TX_CONNECTED handler: replaced the single fixed 2ch/24-bit speaker
     candidate with a candidate-scan loop (mirrors the mic's D-02 pattern). QMX keeps exactly one
     candidate (2ch/24-bit, unchanged behavior). FTX-1 gets a second fallback candidate
     (2ch/16-bit). Winning candidate's bit_resolution is recorded in new static s_spk_bit_resolution
     (reset to 24 in uac_start_with_profile()).
  3. spk_writer_task(): dispatches to dds_render_16bit_stereo() or dds_render_24bit_stereo() based
     on s_spk_bit_resolution, and computes write_bytes dynamically (frames_per_block * 4 for 16-bit
     vs fixed SPK_PUMP_BYTES for 24-bit) so the USB write length always matches the negotiated
     format.
verification: |
  Build verified: idf.py build completed successfully with no new warnings/errors from the changed
  files (only pre-existing unrelated main.cpp unused-function warnings).
  Hardware verification CONFIRMED by user across two separate physical boot cycles against the real
  FTX-1: candidate-scan log shows "Starting spk stream (profile=ftx1_cp210x) candidate 1/2: 48000Hz,
  24-bit, 2ch" fails as expected (hardware genuinely rejects 24-bit, consistent with root cause),
  then "candidate 2/2: 48000Hz, 16-bit, 2ch" succeeds, logging "Speaker pre-opened+claimed". This is
  the first time in the project that the FTX-1 speaker interface has successfully opened. Fix
  confirmed working end-to-end on hardware.

  Known follow-up (separate, not part of this session's fix): with the speaker now successfully
  claiming USB host resources, mic negotiation on the same boot consistently fails with
  "HCD DWC: No more HCD channels available" / "EP Alloc error: ESP_ERR_NOT_SUPPORTED" (reproduced
  identically across 2 power cycles) - likely an ESP32-S3 USB-OTG hardware channel-count ceiling hit
  when both mic and speaker claim resources simultaneously, distinct from the FIFO-lines budget
  tuned in Phase 4. Needs its own debug session.
